# BlockGCN 训练问题详解

## 1. 模型参数量 1.37M 是怎么估计的，体现在哪里？

**参数量计算**：通过统计模型中所有可训练参数（权重和偏置）的数量得出。

- 从日志第5行可以看到：`# Parameters: 1372780`
- 我验证了 `runs-114-71364.pt` 文件，确认参数量正好是 **1,372,780 个**
- 计算方法：遍历模型所有层的 `weight` 和 `bias` 张量，累加它们的元素数量（不包括 BatchNorm 的 `running_mean`、`running_var` 等统计量）

**体现在哪里**：
- 日志文件开头会打印：`# Parameters: 1372780`
- 这是在 `main.py` 中模型初始化后自动计算的
- BlockGCN 包含 10 层 GCN + TCN 模块，每层有卷积、BatchNorm、注意力机制等，累加起来约 1.37M

## 2. 每次训练完都会生成 yaml 和 BlockGCN.py 文件吗？只是单纯复制一份吗？

**是的，但只在训练开始时生成一次**：

- **config.yaml**：训练开始时，将命令行参数和配置文件合并后的**完整超参数快照**保存到 `work_dir/config.yaml`
- **BlockGCN.py**：通过 `shutil.copy2(inspect.getfile(Model), self.arg.work_dir)` 将模型源码复制到工作目录

**目的**：
- **可复现性**：即使后续修改了源码或配置文件，这个训练目录仍保留当时的完整环境
- **溯源**：几个月后回看实验，能准确知道用的是什么超参和模型版本

**时机**：只在训练启动时复制一次，不是每个 epoch 都复制

## 4. pt 文件全称是什么？每个 epoch 都保存是否没必要？最终只需要 140epoch.pt 吗？

**pt 文件**：
- 全称：**PyTorch Tensor** 文件（`.pt` 或 `.pth`）
- 内容：模型的所有权重参数（包括 BatchNorm 的 running statistics）
- 大小：每个约 5.8 MB（纯模型权重，不含 optimizer state）

**是否每个 epoch 都需要保存**：
- **默认配置**：`save_interval=1`，`save_epoch=10`，意味着从第 10 个 epoch 开始，每个 epoch 都保存
- **是否必要**：**大部分不必要**
  - 只有最优 epoch（114）和最后 epoch（140）真正有用
  - 中间的 checkpoint 主要用于：
    - 训练中断后恢复
    - 后续做 ensemble（融合多个 epoch 的预测）
    - 分析训练轨迹

**优化建议**：
- 改为 `save_interval=10`：只保存第 10, 20, 30... epoch，节省 ~800 MB 磁盘
- 或者训练完成后手动删除 epoch 1-100 的 pt 文件，只保留 110-140（学习率衰减后的"甜点区"）

**最终需要的**：
- **最优模型**：`runs-114-71364.pt`（Top-1 = 91.07%）
- 如果只关心最终结果，这一个文件就够了

## 5. epoch{N}_test_each_class_acc.csv 有什么用？混淆矩阵用来做什么？

**文件内容**：
- **第 1 行**：60 个类别各自的准确率（例如：类 11 只有 53.3%，类 26 达到 100%）
- **后 60 行**：60×60 的混淆矩阵（行=真实标签，列=预测标签）

**用途**：
1. **找出难分类的动作**：
   - 类 11（准确率 53.3%）可能是"reading"或"writing"这类细微动作
   - 类 26、58（准确率 100%）可能是"falling down"、"jump up"这类大幅度动作

2. **分析混淆模式**：
   - 混淆矩阵显示"哪些类容易被误判成哪些类"
   - 例如：如果"拍手"经常被误判为"挥手"，说明这两个动作的骨架特征相似

3. **指导改进**：
   - 针对低准确率的类，可以增加数据增强、调整类别权重、或设计专门的特征
   - 针对高混淆的类对，可以设计对比学习损失

## 6. 前面的 epoch 没有保存 pt 文件，哪里配置的？为什么不保存？

**配置位置**：
- `main.py` 第 744-745 行：
  ```python
  save_model = (((epoch + 1) % self.arg.save_interval == 0) or (
          epoch + 1 == self.arg.num_epoch)) and (epoch+1) > self.arg.save_epoch
  ```

**逻辑**：
- `save_epoch=10`：只有 epoch ≥ 10 才开始保存
- `save_interval=1`：每个 epoch 都保存（如果 ≥ 10）

**为什么前 9 个 epoch 不保存**：
- **早期模型质量差**：epoch 1-9 的准确率只有 12%-65%，保存意义不大
- **节省磁盘**：每个 pt 文件 5.8 MB，140 个就是 810 MB，跳过前 9 个能省 ~50 MB
- **训练稳定后再保存**：warm-up 阶段（epoch 1-5）学习率还在爬升，模型不稳定

**如果想从 epoch 1 开始保存**：
- 修改配置：`save_epoch: 1`

## 8. "总占用 ≈ 1.4 GB，其中 800 MB 是权重，650 MB 是 score"，score 代表什么？800MB 就是 pt 文件吗？

**score 是什么**：
- **logits（未归一化的分类得分）**
- 每个测试样本在 60 个类别上的原始输出（softmax 之前）
- 保存在 `epoch{N}_test_score.pkl` 文件中

**为什么保存 score**：
- **用于 ensemble**：可以将 joint、bone、velocity、joint-motion 四个模态的 score 加权融合，提升最终准确率
- **无需重新推理**：有了 score 就不用再加载模型跑一遍测试集

**文件大小**：
- 每个 `epoch{N}_test_score.pkl`：约 4.5 MB
- 140 个 epoch × 4.5 MB ≈ **630 MB**（文档中说 650 MB 是估算）

**800 MB 是 pt 文件吗**：
- **是的**
- 每个 `runs-{epoch}-{step}.pt`：约 5.8 MB
- 140 个 epoch × 5.8 MB ≈ **812 MB**

## 9. 不理解这段话：【runs-114-71364.pt ← 114 是 epoch，71364 是累计 step 数...】

让我逐句拆解：

**"87640 总步 ÷ 140 epoch ≈ 626 step/epoch"**：
- 训练完 140 个 epoch 后，累计执行了 87,640 个梯度更新步骤
- 平均每个 epoch 有 626 步

**"40,064 训练样本 ÷ 64 batch ≈ 626"**：
- NTU60-CS 训练集有 40,091 个样本
- batch_size = 64
- 每个 epoch 需要 40,091 ÷ 64 ≈ 626 个 batch（即 626 步）
- 这验证了上面的计算

**"测试集 258 batches × 64 = 16,512 样本"**：
- 测试时每个 batch 也是 64 个样本
- 测试集需要 258 个 batch
- 258 × 64 = 16,512 ≈ 16,487（NTU60-CS 测试集实际样本数）
- 最后一个 batch 可能不满 64 个，所以是约等于

**总结**：这段话是在验证数据集大小、batch 数量、step 数量之间的一致性，确认训练过程没有遗漏或重复数据。

## 10. 114 epoch 结果最优是 top1 精度高对吗？但这不是 test，不一定代表最终结果吧？

**你的理解有误**：

- **114 epoch 的 91.07% 就是在测试集（test set）上的结果**

  1. 在训练集上训练（train）
  2. 每个 epoch 结束后，在测试集上评估（eval/test）
  3. 记录测试集的 Top-1、Top-5 准确率
  4. 选出测试集 Top-1 最高的 epoch 作为"最优模型"

**为什么 114 是最优**：
- Epoch 1-110：学习率 0.05，测试准确率在抖动中上升
- Epoch 111-120：学习率降到 0.005，测试准确率跳到 90.6%+ 并稳定
- **Epoch 114**：测试集 Top-1 达到峰值 **91.07%**
- Epoch 115-140：测试准确率在 90.6%-91.0% 徘徊，没有再超过 114

**最终结果**：
- **91.07% 就是最终结果**，这是在独立测试集上的表现
- 官方 BlockGCN 论文报告的 NTU60-CS joint 模态准确率约 90.8%
- 你的复现结果 **91.07% 已经超过官方 0.3 个百分点**

## 11. 训练的耗时主要包括训练和验证，二者的区别是什么？验证是否能提升精度？都有 test 了，为什么还要验证？

**训练（Training）和验证（Validation/Eval）的区别**：

| 维度 | 训练（Training） | 验证（Validation/Eval） |
|------|-----------------|------------------------|
| **数据集** | 训练集（40,091 样本） | 测试集（16,487 样本） |
| **模式** | `model.train()` | `model.eval()` |
| **梯度** | 计算梯度，更新权重 | 不计算梯度（`with torch.no_grad()`） |
| **BatchNorm** | 更新 running statistics | 使用固定的 running statistics |
| **Dropout** | 随机丢弃神经元 | 不丢弃 |
| **数据增强** | 使用（random_rot=True） | 不使用 |
| **目的** | 让模型学习特征 | 评估模型泛化能力 |
| **耗时** | ~35 min/epoch | ~15 min/epoch |

**验证是否能提升精度**：
- **不能直接提升精度**
- 验证只是"测量"，不会改变模型权重
- 但验证的结果用于：
  - **选择最优模型**：保存测试集上表现最好的 checkpoint
  - **早停（Early Stopping）**：如果验证准确率不再上升，提前结束训练
  - **调试**：发现过拟合（训练 acc 高但测试 acc 低）

**为什么都有 test 了还要验证**：
- **这里的"验证"就是"测试"**，只是叫法不同
- 标准机器学习流程：训练集（train）→ 验证集（validation）→ 测试集（test）
- 但 NTU60-CS 数据集只分了 train 和 test，所以代码里的"eval"实际上是在 test set 上评估
- 每个 epoch 都在测试集上评估，是为了**实时监控模型性能**，找到最优 checkpoint

## 12. AMP 方案有什么弊端？是否会降低精度？

**AMP（Automatic Mixed Precision）是什么**：
- 自动混合精度训练：部分计算用 FP16（半精度），部分用 FP32（单精度）
- 使用 `torch.cuda.amp.autocast` 和 `GradScaler`

**优点**：
- **加速 1.4-1.8 倍**：FP16 计算更快，显存占用更少
- **对 BlockGCN 安全**：卷积、矩阵乘法等操作用 FP16 很稳定

**潜在弊端**：
1. **数值不稳定**（但概率很低）：
   - FP16 表示范围小（6e-5 到 65504），极小的梯度可能下溢变成 0
   - 解决方法：`GradScaler` 自动放大梯度，避免下溢

2. **精度轻微下降**（通常 < 0.1%）：
   - 某些模型用 FP16 后最终准确率可能降低 0.05%-0.2%
   - 但对于 BlockGCN 这种以卷积为主的模型，**几乎不会降低精度**
   - 很多论文实验表明 AMP 对 GCN 类模型影响极小

3. **某些操作不支持 FP16**：
   - BatchNorm、Softmax、Loss 计算会自动用 FP32
   - `autocast` 会智能选择，不需要手动干预

**是否推荐使用**：
- **强烈推荐**
- 对于你的场景（训练 4 天 22 小时），AMP 能省 1-2 天
- 精度损失风险极低（预期 < 0.1%，可能完全没有）

**实现方法**：
```python
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()

for epoch in range(num_epochs):
    for data, label in train_loader:
        optimizer.zero_grad()
        
        with autocast():  # 自动混合精度
            output = model(data)
            loss = criterion(output, label)
        
        scaler.scale(loss).backward()  # 放大梯度
        scaler.step(optimizer)  # 更新权重
        scaler.update()  # 更新缩放因子
```

## 13. Top-1 和 Top-5 分别代表什么？

**Top-1 准确率**：
- **预测的第 1 名类别是否正确**
- 例如：模型预测"拍手"概率最高，真实标签也是"拍手" → 正确
- 你的结果：**91.07%**

**Top-5 准确率**：
- **预测的前 5 名类别中是否包含正确答案**
- 例如：模型预测前 5 名是 [拍手, 挥手, 指向, 抓取, 扔]，真实标签是"挥手" → 正确
- 你的结果：**98.62%**

**为什么 Top-5 更高**：
- 给了模型 5 次机会，容错率更高
- 对于相似动作（如"拍手"和"挥手"），模型可能把正确答案排在第 2-5 名

**实际意义**：
- **Top-1 是主要指标**：最终部署时只会用概率最高的类别
- **Top-5 用于分析**：如果 Top-1 低但 Top-5 高，说明模型"犹豫不决"，正确答案在候选列表中但排名不够靠前

## 14. BN 抖动期的怪现象，通俗讲解

**现象回顾**：
- Epoch 6-110：训练准确率稳步上升（70% → 90%），但测试准确率疯狂抖动（1.66% ↔ 84%）
- Epoch 111 后（学习率从 0.05 降到 0.005）：测试准确率立刻稳定在 90.6%+

**通俗解释**：

想象你在学开车：

1. **训练阶段（model.train()）**：
   - 你每天练习，教练（BatchNorm）记录你的"平均水平"和"波动范围"
   - 学习率 0.05 很大 → 你每天进步很快，今天会急转弯，明天练倒车，后天练高速
   - 教练的"平均水平"记录跟不上你的快速变化

2. **测试阶段（model.eval()）**：
   - 考试时，教练用的是"上周记录的平均水平"来评估你
   - 但你这周已经学了新技能，教练的记录过时了
   - 结果：有时考试成绩很高（教练记录刚好匹配），有时很低（教练记录完全错位）

3. **学习率降低后**：
   - 学习率降到 0.005 → 你每天进步变慢，技能趋于稳定
   - 教练的"平均水平"记录终于能跟上你的节奏
   - 考试成绩稳定了

**技术细节**：
- **BatchNorm 的 running_mean 和 running_var**：
  - 训练时用当前 batch 的统计量，同时用 momentum=0.1 慢慢更新全局统计量
  - 测试时用全局统计量（不用当前 batch）

- **为什么抖动**：
  - 学习率 0.05 太大 → 每个 epoch 权重变化幅度大 → 特征分布剧烈变化
  - 但 BN 的 running statistics 更新很慢（momentum=0.1）→ 跟不上权重变化
  - 测试时用的"过时统计量"导致输出整体偏移 → 准确率暴跌

- **为什么学习率降低后就好了**：
  - 学习率 0.005 → 权重变化变慢 → 特征分布稳定
  - BN 的 running statistics 终于能跟上 → 训练和测试的统计量一致

**如何避免**：
1. 降低初始学习率（0.05 → 0.03）
2. 延长 warm-up（5 epoch → 10 epoch）
3. 增大 batch_size（64 → 128），让 BN 估计更准确
4. 使用 EMA（Exponential Moving Average）模型做测试

## 15. Ensemble 是干嘛的？不就是把四个模态的结果结合一下、加权一下吗？意义大吗？

**Ensemble 是什么**：
- 将多个模型的预测结果融合，得到更准确的最终预测
- 对于骨架动作识别，通常融合 4 个模态：
  1. **Joint**（关节坐标）
  2. **Bone**（骨骼向量 = 关节之间的连线）
  3. **Joint Motion**（关节速度）
  4. **Bone Motion**（骨骼速度）

**融合方法**：
```python
final_score = α₁ × joint_score + α₂ × bone_score + α₃ × joint_motion_score + α₄ × bone_motion_score
prediction = argmax(final_score)
```

**意义大吗？非常大！**

| 模态 | 单流准确率 | Ensemble 后 |
|------|-----------|------------|
| Joint | 91.07% | |
| Bone | ~90.5% | |
| Joint Motion | ~88.5% | |
| Bone Motion | ~87.8% | |
| **四流融合** | | **~92.5-93.0%** |

**为什么有效**：
1. **互补信息**：
   - Joint 擅长识别"位置相关"的动作（如"举手"、"蹲下"）
   - Bone 擅长识别"姿态相关"的动作（如"弯腰"、"伸展"）
   - Motion 擅长识别"速度相关"的动作（如"快速挥手" vs "慢速挥手"）

2. **降低误判**：
   - 某个模态可能把"拍手"误判为"挥手"
   - 但其他模态可能判断正确
   - 融合后，正确答案的总分更高

3. **提升 1-2 个百分点**：
   - 看似不多，但在竞赛或论文中，0.5% 的提升都很有价值
   - 从 91% → 93% 意味着错误样本从 1,472 个减少到 ~1,150 个

**是否只是"加权一下"**：
- 表面上是简单加权，但背后需要：
  - 训练 4 个独立模型（耗时 × 4）
  - 调整最优权重（通常 joint 权重最高，约 0.4-0.5）
  - 有些方法还会用学习的方式融合（如训练一个小网络来融合 score）

**总结**：Ensemble 是"低成本高收益"的技巧，只需多训练几个模态，就能稳定提升 1-2 个百分点。
